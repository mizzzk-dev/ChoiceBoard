# ChoiceBoard 実装着手用チケット分解（M1〜M2）

## 1. 全体方針

### 実装順の基本方針
- **M1（比較体験の最小成立）を先に完了**し、M2（保存・共有と信頼性補強）はM1の成果物に積み上げる。
- 先に作るべき基盤は、`data（スキーマ） -> backend（取得/抽出/正規化API） -> frontend（入力/結果表示） -> qa（回帰/E2E）` の順。
- 後回しにしてよいものは、LLM要約・共有URL・セル単位出典表示（いずれもM2）。
- クリティカルパスは以下。
  1. M1データモデル確定
  2. compare API（取得→抽出→正規化）
  3. 結果画面表示
  4. M1 E2E通過
  5. share API + snapshot保存
  6. summarize API（事実入力限定）
  7. 共有画面 + M2 E2E通過

### 実行原則
- チケット粒度は **0.5〜2日**。
- 仕様凍結済み範囲のみ実装し、追加機能提案は行わない。
- 各チケットに完了条件（DoD）を必須化する。
- 依存関係を明確にし、並列可能箇所を明示する。

## 5. 実装着手前の注意点

### 詰まりやすいポイント
- 価格文言のゆらぎ（`$29/month`, `per user`, `from` など）で正規化が破綻しやすい。
- HTML構造差で抽出率が急落しやすい。
- summarizeで事実外テキストが混入しやすい。

### 先に決めておくべき実装ルール
- status遷移の単一責務（API層のみ更新）。
- unknown優先ルール（推測補完禁止）。
- confidence判定条件の固定（根拠件数/抽出経路）。
- エラーコード命名規約（入力/取得/解析/共有/要約）。

### テスト優先順位
1. 正規化ユニット（最優先）
2. compare/share API統合
3. E2E（M1導線→M2導線）
4. 要約整合（ガードレール）

### モックと本実装の切り分け
- 外部ページ取得はテスト時のみモックし、アプリ本体の処理フローは本実装を通す。
- LLMは失敗時縮退を本実装し、モック要約で成功扱いにしない。
- UIはAPIモックで先行可能だが、M1/M2ゲート前に必ず実API接続で確認する。
