# ChoiceBoard MVP 実装前意思決定ドキュメント

## 1. 今回の前提整理

- プロダクト名は **ChoiceBoard**。
- 比較対象は **SaaS料金ページURL最大3件**。
- MVPで返す成果物は、(1)正規化比較表 (2)主な違いの要約 (3)向いている人・チーム (4)注意点 (5)保存・共有ページ。
- 技術スタックは **Next.js / TypeScript / PostgreSQL / Prisma / Tailwind**。
- UX方針は「AIチャット」ではなく、**比較して選ぶためのツール**として設計する。
- 品質方針は「事実ベース優先」。未確認情報は事実として扱わない。
- 抽出・正規化は決定論的処理を主軸にし、LLMは要約・整理など補助用途に限定する。

## 2. 未確定事項6点

### 論点1: 対応対象ページ範囲（取得対象の境界）

1. **論点名**  
   対応対象ページ範囲（静的HTML中心か、JS動的ページまで含むか）

2. **なぜ今決める必要があるか**  
   抽出パイプライン、タイムアウト、失敗時UX、実装コストが最初に大きく分岐するため。

3. **選択肢A / B / C**  
   - A: 静的HTML中心のみ対応（SSR後HTML取得ベース）
   - B: 静的HTML + 一部JSレンダリング（ヘッドレス実行あり）
   - C: JS依存ページを含む広範囲対応

4. **各選択肢のメリット・デメリット**  
   - A
     - メリット: 実装が最小、再現性が高い、運用コストが低い
     - デメリット: 取得不可ページが増える
   - B
     - メリット: カバー率向上
     - デメリット: 実装とインフラが複雑化、MVP期間が伸びる
   - C
     - メリット: 対応範囲最大
     - デメリット: MVPとして過剰、安定化が難しい

5. **MVPとしての推奨案**  
   **A: 静的HTML中心のみ対応**

6. **推奨理由**  
   最短で比較体験を成立させるには、抽出精度と再現性を優先するべきであり、対応範囲を限定したほうが品質担保しやすい。

7. **将来拡張時に見直す余地**  
   失敗率データを収集後、対象業種でJS依存が多い場合にBへ段階拡張する。

3. **選択肢A / B / C**  
   - A: USD / JPY のみ + 英語ページ中心
   - B: 主要通貨（USD/EUR/GBP/JPY）+ 英語・日本語
   - C: 通貨・言語制限なし

4. **各選択肢のメリット・デメリット**  
   - A
     - メリット: 正規化仕様が明確、テスト容易

   - B
     - メリット: 利用可能範囲が広い
     - デメリット: 例外処理が増えMVPの密度が落ちる
   - C
     - メリット: 制約が少ない

## 3. 6点に対する推奨意思決定（確定）

以下をMVPの固定意思決定とする。

1. 対象ページは **静的HTML中心のみ**。JS依存ページは「非対応または要確認」として返す。  
2. 通貨・言語は **USD/JPY + 英語中心**（日本語ページは試験対応）。  
3. 価格比較は **全抽出プラン並列表示**（同等プラン推定はしない）。  
4. 価格状態は **Free / Trial / Contact Sales / Unknown** の固定4カテゴリ。  
5. 共有ページは **スナップショット固定保存**。  
6. confidenceは **High/Medium/Low** の3段階表示。

## 4. 意思決定を反映した Milestone 1 仕様凍結

> Milestone 1 定義: **比較体験の最小成立**

### 4-1. スコープ

**入れる**
- URL入力（1〜3件）
- 入力バリデーション（URL形式、重複警告、件数上限）
- サーバー側HTML取得（静的HTML中心）
- 決定論的抽出（プラン名/価格/課金周期/価格状態）
- 正規化比較表の表示
- 抽出不能項目の明示（Unknown）
- ジョブ単位の一時保存（結果画面表示に必要な最小データ）

**入れない**
- 共有URL公開
- LLM要約（主な違い/向いている人）
- 高度な機能比較（SSO/API制限など網羅）
- JSレンダリング依存ページ対応

### 4-2. ユーザーフロー

1. ユーザーが `/` でURLを最大3件入力し、比較実行。  
2. `POST /api/compare` でジョブ作成、処理中表示へ遷移。  
3. サーバーで取得→抽出→正規化。  
4. 完了後 `/result/{jobId}` を表示。  
5. ユーザーは比較表と注意点（取得不可/曖昧）を確認。

### 4-3. 機能仕様

#### 画面一覧
- `/`（入力画面）
- `/result/{jobId}`（結果画面）
- エラー状態表示（入力エラー/取得失敗/解析失敗）

#### 各画面の責務
- 入力画面: URL入力、バリデーション、比較開始
- 結果画面: 正規化比較表、confidence表示、注意点表示

#### 主要コンポーネント
- `UrlInputForm`
- `ProcessingStatus`
- `ComparisonTable`
- `ConfidenceBadge`
- `CautionPanel`

#### 主要API/サーバー処理
- `POST /api/compare`: ジョブ作成と同期処理開始
- `GET /api/compare/{jobId}`: 結果取得
- サーバー内部処理
  - URL正規化
  - HTML fetch（タイムアウト・サイズ上限）
  - DOM抽出ルール適用
  - 通貨/価格/周期正規化

#### DBで保存するもの / しないもの
- 保存する
  - Job基本情報（status, createdAt）
  - Source URLとfetchメタ
  - 抽出済みプラン・価格・分類・confidence
  - 注意点（取得失敗理由）
- 保存しない
  - 生HTML全文（M1では保管しない、ハッシュのみ）
  - 共有トークン

### 4-4. データ仕様

#### 入力データ
- `inputUrls: string[1..3]`
- `requestedAt`

#### 正規化データ
- `serviceName`
- `planName`
- `priceAmount`
- `currency` (USD|JPY|Unknown)
- `billingCycle` (monthly|annual|unknown)
- `priceState` (free|trial|contact_sales|unknown)
- `confidence` (high|medium|low)

#### 比較結果データ
- サービスごとのプラン配列
- 共通列（価格、周期、価格状態、備考）
- 抽出不能フィールド一覧

#### 保存ページに必要なデータ
- M1では未対応（jobId表示のみ）

### 4-5. 品質ルール

- 未確認データは必ず `Unknown` と表示。
- confidenceは抽出根拠に基づく3段階で統一。
- 出典表示はM1で最小限（元URL単位）とし、セル単位出典はM2で実装。
- エラー時は失敗種別を明示（入力不正 / 取得不可 / 解析不可）。
- モックデータで見た目のみ成立させる実装は禁止。最低1件は実URLでE2E成立させる。

### 4-6. 完了条件

- 1〜3URL入力から結果表示まで完走できる。
- 比較表に価格・周期・価格状態が表示される。
- 未取得値がUnknownで表示される。
- 主要失敗ケースでユーザー向けエラー文言が出る。
- QA基準: 固定テストURLセットで成功率基準（例: 70%以上）を満たす。

### 4-7. 実装塊（チケット分解用）

- **frontend**: 入力画面、処理中UI、結果テーブル、注意点UI
- **backend**: compare API、fetch/parse/normalizeパイプライン、エラーハンドリング
- **data**: Job/Source/Plan系スキーマ、最小インデックス
- **qa**: URLバリデーション試験、抽出回帰、E2E（入力→結果）

## 5. 意思決定を反映した Milestone 2 仕様凍結

> Milestone 2 定義: **保存・共有と信頼性の補強**

### 5-1. スコープ

**入れる**
- 共有URL発行（スナップショット固定）
- 結果ページの再表示（同一トークンで同一内容）
- セル単位または項目単位の出典表示
- 注意点の強化（曖昧/非対応/要確認の理由明記）
- LLM補助による「主な違い」「向いている人」生成（事実入力限定）

**入れない**
- 閲覧時の再クローリング
- 同等プラン自動マッピング
- 定期再取得スケジューラ

### 5-2. ユーザーフロー

1. M1と同様に比較実行。  
2. 結果画面で「保存して共有」を実行。  
3. `POST /api/share/{jobId}` で shareToken 発行。  
4. `/share/{token}` を第三者が閲覧。  
5. 表示内容は生成時スナップショットに固定。

### 5-3. 機能仕様

#### 画面一覧
- `/result/{jobId}`（共有作成導線追加）
- `/share/{token}`（共有閲覧）

#### 各画面の責務
- 結果画面: 保存操作、共有URLコピー、要約表示
- 共有画面: 比較表・要約・注意点・出典閲覧（編集不可）

#### 主要コンポーネント
- `ShareButton`
- `ShareLinkPanel`
- `EvidenceLinkList`
- `DiffSummaryCard`
- `FitRecommendationCard`

#### 主要API/サーバー処理
- `POST /api/share/{jobId}`: スナップショット保存 + token発行
- `GET /api/share/{token}`: 共有結果取得
- `POST /api/summarize/{jobId}`: 事実テーブル入力で要約生成
- サーバーガードレール
  - LLM入力は抽出済み事実のみ
  - 根拠なし文を除外

#### DBで保存するもの / しないもの
- 保存する
  - shareToken
  - 共有スナップショットJSON
  - 要約本文 + 使用モデル + promptVersion
  - evidence参照（sourceUrl, path or selector）
- 保存しない
  - LLMへの生プロンプト全文（M2では要約版メタのみ）

### 5-4. データ仕様

#### 入力データ
- M1入力 + `shareRequestedAt`

#### 正規化データ
- M1項目 + `evidenceRef`

#### 比較結果データ
- `diffSummary`
- `fitSummary`
- `cautions[]`
- `evidenceMap`

#### 保存ページに必要なデータ
- `shareToken`
- `snapshotVersion`
- `snapshotCreatedAt`
- `comparisonPayload`
- `summaries`
- `evidences`

### 5-5. 品質ルール

- 未確認データは共有画面でもUnknown表記を維持。
- confidenceは共有画面でも必須表示。
- 出典は最低「元URL + 対応項目」を表示。
- エラー時
  - 共有作成失敗: 再試行導線
  - 要約失敗: 比較表のみ表示（機能縮退）
- モック要約は禁止。LLM不通時は要約セクションを「生成失敗」と明示。

### 5-6. 完了条件

- 共有URLで同一結果が再表示される。
- 出典情報が要約・比較表の主要項目に紐づく。
- LLM要約が事実テーブルにない情報を含まない。
- 共有URL閲覧で比較判断に必要な情報（表・要約・注意点）が完結する。

### 5-7. 実装塊（チケット分解用）

- **frontend**: 共有導線、共有閲覧画面、要約カード、出典UI
- **backend**: share API、snapshot保存、summarize API、LLMガードレール
- **data**: Share/Summary/Evidence系スキーマ拡張、snapshot versioning
- **qa**: 共有再現性テスト、要約事実整合テスト、縮退動作テスト
